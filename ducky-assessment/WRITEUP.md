# WRITEUP.md — Ducky: AI Usage Tracker

## 1. Tracking Approach: What Signals and Why

### Signal 1: Running Processes (`/daemon/collectors/processes.js`)
**What:** Poll the process list every 5 seconds using `tasklist` (Windows) or `ps` (POSIX) and match against a curated list of known AI tool process names: `cursor`, `copilot`, `ollama`, `claude`, `tabnine`, `codeium`, and others.

**Why:** The most direct signal available. If a developer has Cursor IDE or the GitHub Copilot Language Server running, that is unambiguous evidence of AI tooling installed and active. Process presence also persists across editing sessions — it doesn't require observing individual keystrokes.

**Signal value:** High. Presence of an AI IDE or language server is a strong leading indicator of AI-assisted development, regardless of how the tool is actually used in a given session.

---

### Signal 2: Network Connections (`/daemon/collectors/network.js`)
**What:** Poll active TCP connections every 10 seconds using `netstat`/`ss`, resolve known AI API hostnames (`api.openai.com`, `api.anthropic.com`, `copilot-proxy.githubusercontent.com`, etc.) to IPs, and match.

**Why:** Network connections to AI inference endpoints are the most definitive signal available without instrumenting the editor itself. A connection to `api.openai.com:443` during a coding session means an API call was made to OpenAI — there is no false-positive scenario where this happens without AI usage.

**Signal value:** Highest possible. It directly observes AI API calls in-flight. DNS resolution is re-attempted each poll to handle IP rotation.

**Privacy note:** We log only host, IP, and TCP state — never payload contents.

---

### Signal 3: File Change Velocity and Batch Size (`/daemon/collectors/filesystem.js`)
**What:** Watch the project directory with `chokidar`. Group file change events within 2-second windows into batches. Flag batches of ≥5 simultaneous file changes as high-probability AI autocomplete events.

**Why:** Human developers edit files sequentially — one file at a time, incrementally. AI autocomplete tools (Copilot, Cursor, Codeium) generate large multi-line or multi-file completions that land as a burst of `write` events within milliseconds of each other. This pattern is structurally different from human typing and measurable.

**Signal value:** Medium-high. Large batches have low false-positive rates in isolation; combined with process detection, they become very reliable.

---

### Signal 4: Clipboard Metadata (`/daemon/collectors/clipboard.js`)
**What:** Poll the system clipboard every 3 seconds (PowerShell on Windows, `pbpaste` on macOS, `xclip` on Linux). Do NOT store clipboard contents. Record only: character count, line count, whether it looks like code (regex heuristic), and whether it looks like AI-generated code (large + structured).

**Why:** A common AI workflow pattern is: generate in ChatGPT/Claude web → copy → paste into editor. This leaves a trace: a large, well-formatted block of code appearing in clipboard that was not typed. The timing correlation between a clipboard paste event and a file change event is a strong compound signal.

**Signal value:** Medium. High-value when correlated with file changes at the same timestamp. Privacy-safe by design — we never write clipboard contents to disk.

---

### Signal 5: Git Commit Patterns (`/daemon/collectors/git.js`)
**What:** Poll `git log` every 15 seconds for new commits. Extract diff stats (files changed, insertions, deletions) and apply heuristics to flag AI-assisted commits: large insertions in many files at once, or commit messages matching Conventional Commits format (which AI tools often generate automatically).

**Why:** Human commits tend to be incremental and focused. AI-assisted commits often involve large diffs touching many files simultaneously, or suspiciously perfect Conventional Commits messages generated by the AI. The ratio of insertions-to-files-changed is a particularly reliable indicator.

**Signal value:** Medium. Strong when combined with other signals. False-positive risk exists (skilled engineers can make large commits legitimately), so it's treated as corroborating evidence, not primary signal.

---

### Signal 6: Editor Artifacts and Extensions (`/daemon/collectors/editor.js`)
**What:** One-shot scan at startup of the home directory, project root, and VS Code extensions directory for well-known AI tool configuration directories (`.cursor/`, `.copilot/`, `.continue/`, `.codeium/`) and installed VS Code extension IDs.

**Why:** The presence of AI tooling infrastructure is informative even if the tool wasn't actively used during the session. A developer who has GitHub Copilot installed and active in VS Code has self-selected into the AI tooling ecosystem.

**Signal value:** Low as a standalone signal (tool installed ≠ tool used), but high as a baseline context signal that amplifies other signals.

## 2. Why Tracking AI Usage Reveals What Traditional Assessments Miss

Traditional technical interviews and take-home assessments evaluate a static snapshot: can the candidate produce correct code under artificial constraints? This misses the most important dimension of modern software engineering: **how does the developer think alongside their tools?**

AI tooling has stratified developer productivity along a new axis. Two developers might both produce a working solution, but one uses AI as a force multiplier — generating scaffolding, catching edge cases, refactoring fearfully — while the other types every line manually. The former is not "cheating"; they are demonstrating a critical professional skill: **effective AI orchestration under engineering constraints**.

What ducky reveals that traditional assessments cannot:

1. **Tool awareness**: Does the developer know which AI tools exist and have them configured? This is a leading indicator of staying current with the field.

2. **Workflow integration**: Are AI APIs being called during active coding (network signals)? Or is the developer ignoring available tooling?

3. **Code generation patterns**: Do file change bursts coincide with AI process activity? This shows the developer is actively invoking and integrating AI completions rather than manually typing everything.

4. **Verification behavior**: Does the developer commit AI-generated code immediately (large diffs, single commit) or incrementally (many small commits, smaller diffs)? The latter suggests critical evaluation of AI output — a far more valuable signal.

5. **Research patterns** (clipboard signal): Is the developer copying from AI chat interfaces and adapting the output, or generating directly in-editor? Both are valid, and the distinction is meaningful.

## 3. Limitations and Extensions

### Current Limitations

- **No intra-editor visibility**: We cannot distinguish between AI-generated and human-typed text at the keystroke level without a custom editor extension or language server proxy. The file-change batching heuristic is an approximation.

- **Encrypted traffic**: Network connections to AI APIs are TLS-encrypted. We detect connection presence, not API call content or frequency within an established connection.

- **Clipboard polling latency**: At 3-second intervals, we may miss fast copy-paste cycles. True clipboard monitoring would require OS-level hooks (WinAPI `SetClipboardViewer` on Windows, `NSPasteboard` notifications on macOS).

- **Process name collisions**: A process named `claude` could be something else. The heuristic is probabilistic.

### Extensions With Unconstrained Access

**1. VS Code Extension (if allowed by assessor)**
A VS Code extension would give us access to:
- `vscode.workspace.onDidChangeTextDocument` — exact bytes added/removed, cursor position, character-by-character insertion velocity
- `vscode.commands.executeCommand('editor.action.inlineSuggest.commit')` events — direct confirmation of Copilot suggestion accepted
- Document diff at accept-time — exact provenance of each code block

This would eliminate heuristics entirely and provide ground truth.

**2. Language Server Protocol (LSP) Proxy**
Intercept LSP messages between the editor and any AI language server by running a transparent proxy on the LSP socket. This gives exact completion request/response pairs, latency, and acceptance rates without requiring editor plugin access.

**3. GitHub Copilot Telemetry API**
GitHub exposes Copilot usage telemetry to organization admins via the Copilot API (`/orgs/{org}/copilot/usage`). This gives: suggestions shown, suggestions accepted, lines of code accepted — exact ground truth for Copilot usage. Requires org admin access.

**4. eBPF System Call Tracing (Linux)**
Using `bpftrace` or `ebpf`, we can hook `read()`/`write()` syscalls on editor processes and correlate clipboard operations, file writes, and network I/O at the kernel level. This gives microsecond-precision correlation between AI API calls and file write events — the strongest possible signal without full source access.

**5. Browser History Correlation**
On developer machines with local Chrome/Firefox profiles, browser history files (SQLite databases at known paths) can be queried for visits to `chat.openai.com`, `claude.ai`, `perplexity.ai`, and similar, correlated with coding session timestamps. This reveals the "tab switch to ChatGPT, copy answer, paste into editor" pattern.

**6. Statistical Baseline Modeling**
With multiple ducky sessions from the same developer, build a personal baseline of their normal file-change velocity, commit size distribution, and clipboard usage patterns. Deviations from that baseline during an assessment session are stronger signals than absolute thresholds, which vary by individual.
